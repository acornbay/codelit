How to Share Code on Twitter, LinkedIn & Instagram

By Anmol Dubey 8 min read January 2025

In this article

  2. Step-by-step: code to shareable image
  3. Twitter / X — exact settings
  4. LinkedIn — what works best
  5. Instagram — creative approach
  6. GitHub README embeds

Why raw code text fails on social media

When you paste raw code directly into a tweet or a LinkedIn post, three things happen: the platform collapses it into a wall of uniform text, the syntax colouring disappears completely, and the indentation often gets mangled. The result looks like a terminal dump — fine for a colleague who already understands the context, but useless for anyone scrolling past.

Code images solve all three problems simultaneously. They preserve your syntax colours exactly, they keep your indentation intact, and — crucially — they render as a visual card in the feed rather than a block of text. A well-designed code image stops the scroll in the same way a clean diagram or a striking photo does. Developers report 2–4x higher engagement on posts that use formatted code images compared to pasted text, because the image itself signals quality and effort before anyone reads a single line.

There's also a practical benefit: images are indexed and shared differently from text. A code image that gets retweeted carries your screenshot — and implicitly your branding — with it. A text paste just gets copy-pasted elsewhere with no trace of the original author.

Step-by-step: from code to shareable image in CodeHilite

  1. Open CodeHilite at codehilite.xyz. No account or sign-up needed — the app loads instantly in your browser.
  2. Paste your code into the editor on the left panel. CodeHilite uses highlight.js to automatically detect your programming language based on syntax patterns. If it gets it wrong (common with short snippets), use the language dropdown to select the correct one manually.
  3. Choose your theme. The theme controls both the background colour of the code card and the syntax colouring. This single choice has the biggest impact on how your image looks in a feed — see the themes article below for detailed recommendations.
  4. Set your background. This is the area surrounding the code card. Solid white gives a clean, minimal look. Gradients add visual drama that performs better on Instagram. Transparent is ideal if you plan to composite the image in Figma or Canva.
  5. Adjust the card width. Use the width slider to control how many characters fit per line. Wider cards show more code but may appear small on mobile feeds. Narrower cards force line breaks but scale better on all devices.
  6. Turn on the window titlebar (optional). The macOS-style titlebar with red/yellow/green dots gives your screenshot a polished, contextual feel. You can also set a custom filename — e.g. auth-middleware.ts — which adds useful context for viewers.
  7. Enable line numbers if your post will reference specific lines (e.g. "look at line 12"). Disable them if you want a cleaner look for short snippets.
  8. Click Export → PNG. For social media, PNG is almost always the right choice. It's lossless, transparent-background-capable, and renders sharply at any size. See the export article below for a full format breakdown.

💡 Pro tip — keep snippets short: Social media is a skimming medium. Aim for 10–20 lines of code maximum. If you need to show more, crop to the most interesting part and explain the rest in your caption. A beautiful 15-line snippet will always outperform a scrollable 80-line one.

Twitter / X — exact settings that work

Twitter's image compression is aggressive — it converts most uploads to JPEG internally, which can blur text. To counteract this, export your code image at the standard size (CodeHilite's default works well) and always use PNG. Twitter tends to preserve PNG quality better than JPEG.

Recommended settings for Twitter

ThemeAtom One Dark or Dracula — dark themes render better in night mode and stand out in light-mode feeds
BackgroundGradient or solid dark — avoid pure white as it can disappear against Twitter's white background
Font size14px — readable when Twitter scales the preview thumbnail
Card width560–620px — matches Twitter's image preview width well
Line numbersOff for short snippets, on if you reference specific lines in the tweet text
Export formatPNG

Twitter images display at 16:9 ratio in the feed. If your code card is taller than it is wide (portrait orientation), Twitter will letterbox it with grey bars. Aim for a roughly landscape or square crop — increase padding or add extra blank lines at the end of your snippet to nudge the aspect ratio into landscape territory if needed.

LinkedIn — what actually performs well

LinkedIn's audience skews professional, and the algorithm rewards content that looks like it belongs to a thought leader rather than a casual tweet. Code images on LinkedIn should feel polished and credible, not edgy or dark.

Recommended settings for LinkedIn

ThemeGitHub Light or Visual Studio — light themes feel corporate and professional; they also display well on LinkedIn's white background
BackgroundWhite or very light grey — keeps focus on the code itself
Font size15–16px — slightly larger for readability in LinkedIn's feed
Card width700–800px — LinkedIn shows wide images well, especially in desktop view
Window titlebarOn, with a descriptive filename — adds context and looks intentional

On LinkedIn, pair your code image with a meaningful caption that explains why the snippet is interesting. "Here's a cleaner way to handle async errors in Node.js:" followed by the image works far better than posting the image alone. The algorithm also favours native document posts (PDFs), so consider exporting your snippet as a PDF and uploading it as a carousel slide.

Instagram — a different strategy

Instagram is primarily a visual platform and code is not inherently visual — but code images can absolutely perform well on developer-focused accounts when you lean into the aesthetic rather than the technical content.

On Instagram, your code image competes with photography and design. That means bold gradient backgrounds, large font sizes, and strong visual contrast matter much more than on Twitter or LinkedIn. Treat the code image less as a technical resource and more as a visual design piece that happens to contain code.

Recommended settings for Instagram

BackgroundBold gradient (purple-to-pink, blue-to-teal) — makes it visually striking against a scrolling feed
ThemeDracula or Nord — vibrant but readable on gradient backgrounds
Font size16–18px — Instagram crops thumbnails; larger text remains readable at thumbnail size
Aspect ratioSquare (1:1) for feed posts — adjust padding to achieve a roughly square output
Snippet length5–12 lines maximum — Instagram viewers spend 1–2 seconds on each post

GitHub README embeds

README files support Markdown image embeds. After exporting your code snippet as a PNG, you can either upload it directly to your repository and reference it with a relative path, or host it externally and embed it with an absolute URL.

![Auth middleware code snippet](./assets/auth-snippet.png) ![Code snippet](https://user-images.githubusercontent.com/…/auth-snippet.png) <p align="center"> <img src="./assets/auth-snippet.png" alt="Auth middleware" width="600"> </p>

For README embeds, use GitHub Light theme to match GitHub's default rendering. Add a descriptive alt attribute — it helps with accessibility and appears as fallback text if the image fails to load. Keep the image width under 800px to avoid horizontal scrolling on mobile.

Writing the caption that makes people click

The code image is the hook; the caption is the payoff. A good caption does three things: it tells people why the snippet is interesting, what problem it solves, and what they'll learn by reading it. Vague captions like "cool trick 🔥" get ignored. Specific captions like "3 lines to eliminate nested if-else chains in Python — here's how:" get saves and shares.

Always include a question at the end to drive comments — "How do you handle this in your codebase?" costs nothing to add and can double your engagement. The algorithm on every platform rewards comments more than likes.

💡 Content formats that consistently perform well: "I refactored this in X lines instead of Y", "Here's the bug I spent 3 hours finding", "The difference between X and Y explained in code", "This one-liner replaces 20 lines of boilerplate". These frames give viewers an immediate reason to keep reading.

Best Syntax Highlighting Themes for Code Screenshots

By Anmol Dubey 6 min read January 2025

The syntax theme you choose is the single biggest factor in how your code screenshot looks — more than font, more than background, more than padding. Pick the wrong theme and your screenshot looks like a console window. Pick the right one and it looks like a deliberate design choice.

CodeHilite ships with 16+ themes, and each one has a different personality and ideal use case. Here's a detailed breakdown of every major theme and exactly when to reach for it.

Dark themes

Dark themes dominate on social media because they create natural contrast and look good against both white and dark platform backgrounds. They're also easier on the eyes for longer code snippets.

Atom One Dark — the best all-rounder

Atom One Dark is the most popular theme for code screenshots worldwide, and for good reason. Its colour palette is carefully balanced: muted greens for strings, soft blues for keywords, and subdued purples for types — nothing screams, but everything is readable. The background is a dark neutral grey (#282c34) that pairs beautifully with almost any background colour or gradient you put behind the card.

Use Atom One Dark when you want your code to look professional and clean, particularly for Twitter, LinkedIn, and documentation screenshots. It's the "always safe" choice.

Dracula — maximum visual impact

Dracula's signature is its vivid, saturated colours: bright pink for keywords, warm yellow for strings, cyan for functions. It's one of the most visually striking themes available and performs extremely well on Instagram and in slide decks where you want the screenshot to grab attention from across the room.

The downside: Dracula is busy. For long snippets (30+ lines), the vibrant colours can become visually fatiguing. Use it for short, punchy snippets where you want maximum visual impact.

Tokyo Night — modern and refined

Tokyo Night uses cool blues and soft purples to create a sophisticated, almost melancholy aesthetic. It's become popular among developers who want something distinctive that isn't Dracula or Atom One Dark. The background is a deep blue-grey (#1a1b2e) that looks particularly striking when paired with a dark blue or indigo gradient background.

Great for developers with a consistent personal brand built around blues and cool tones.

Nord — documentation-first

Nord's arctic colour palette (cool greys, desaturated blues, muted teals) was designed specifically for readability. It's easy on the eyes for extended reading and creates a calm, professional impression. It's the best dark theme for documentation screenshots and tutorials where you want to minimize visual distraction.

Nord is also particularly effective for screenshots that will be printed or embedded in PDFs, because its muted tones translate well to greyscale.

Night Owl — designed for accessibility

Night Owl was designed by Sarah Drasner with colour blindness accessibility in mind. Its colours are carefully chosen to remain distinguishable even for viewers with common types of colour blindness. If your content reaches a broad audience and accessibility matters to you, Night Owl is an excellent choice.

Monokai — classic developer aesthetic

One of the oldest and most recognisable themes in development. Monokai's bright colours on a very dark background (#272822) create a high-contrast, punchy look that feels distinctly "developer". It's particularly well-suited to language-specific content targeting an audience of experienced developers who'll recognise the theme.

Light themes

Light themes work best on platforms with white backgrounds, in documentation, and in any context where the code needs to feel approachable rather than edgy. They're also more legible when printed.

GitHub Light — the universal standard

GitHub Light is what millions of developers see code in every day on GitHub.com. Using it for your screenshots signals familiarity — viewers immediately understand the context without thinking about it. It's the best choice for README embeds (since it matches GitHub's own syntax highlighting), documentation screenshots, and LinkedIn posts targeting engineering managers and recruiters.

Visual Studio — corporate and professional

Visual Studio's light theme is clean, spacious, and deliberately neutral. It's the safest light theme for professional contexts and works particularly well on LinkedIn and in investor decks, technical proposals, or any document where you want the code to feel authoritative rather than creative.

Solarized Light — warm and readable

Solarized's distinctive warm beige (#fdf6e3) background gives screenshots an almost printed-book quality. It's less common than GitHub Light, which makes it distinctive — particularly good if you want your screenshots to have a recognisable personal style. Works especially well in blog posts and educational content.

Xcode — ideal for Apple platform content

If you're creating content about Swift, SwiftUI, iOS, or macOS development, Xcode Light's colour palette will immediately resonate with your audience — it's exactly what they see every day in their IDE. A small but powerful signal of authenticity.

Choosing your background

The background (the area outside the code card) is as important as the theme. Here are the combinations that work best:

  • Dark theme + gradient background: The most visually striking combination. Deep purple-to-blue gradients with Atom One Dark, coral-to-pink with Dracula, teal-to-blue with Nord.
  • Light theme + white background: Clean and distraction-free. Best for documentation and LinkedIn.
  • Dark theme + solid dark background: Very minimal and elegant. The card floats in space.
  • Light theme + light gradient: Subtle and refined. Works well for portfolios and personal sites.

💡 Build a consistent brand: Pick one dark theme and one light theme and stick with them across all your content. Consistent visual styling makes your posts instantly recognisable in a feed and builds audience expectation over time. Many developer influencers are immediately identifiable purely from their code screenshot aesthetic.

Font recommendations

CodeHilite's default font is JetBrains Mono — an excellent choice that renders cleanly at all sizes and includes ligatures (characters like !=, =>, and ->> are rendered as single symbols for better readability). Other good choices available in CodeHilite:

  • Fira Code — similar to JetBrains Mono with slightly different ligatures; very popular for code screenshots
  • Source Code Pro — Adobe's clean monospace, no ligatures, very legible at smaller sizes
  • Cascadia Code — Microsoft's entry, slightly more playful than JetBrains Mono

Avoid fonts that aren't designed for code (like Courier New) — they render poorly at screenshot resolutions and look dated. Stick to purpose-built monospace fonts for the sharpest output.

Level Up Your Developer Portfolio With Code Images

By Anmol Dubey 7 min read February 2025

When a recruiter or potential client lands on your portfolio, they make a snap judgement in 5–10 seconds. That judgement is almost entirely visual — it happens before they've read a line of your bio or clicked any of your projects. Code images are one of the most powerful ways to upgrade that first impression, because they signal that you care about presentation — and presentation is one of the most transferable markers of code quality.

A developer who presents their code beautifully is often (rightly or wrongly) assumed to write beautiful code. It's a shortcut heuristic, but it's a real one. This guide covers every place you should use code images in your portfolio and the exact approach for each.

Where to use code images

Project cards and thumbnails

Most developer portfolios use a project's UI screenshot as the card thumbnail. But UI screenshots all look similar — they're rectangles of interface that blur together. A code snippet thumbnail is immediately distinctive and often more informative about what you actually did technically. Show a key function, algorithm, or hook from each project instead of (or alongside) a UI screenshot.

Pick the snippet that best demonstrates your technical decision-making — not the longest or most complex one, but the one that someone who knows your field would look at and say "oh, that's a smart approach".

About / Skills section

Many portfolios list skills as a row of logo icons. This is fine but impersonal. Instead, consider showing a short snippet in your primary language that demonstrates your actual writing style. A 10-line function says more about how you think than a list of framework logos. It's concrete evidence versus claimed expertise.

Case studies and project deep-dives

If you write in-depth case studies about your projects (you should — they differentiate you enormously), code images are essential supporting material. Walk through your architectural decisions with annotated snippets. Use the line highlight feature to draw attention to the specific part you're discussing.

A good case study structure: problem statement → approach → key implementation decision (with code image) → result/metrics. The code image is what makes your case study a technical artefact rather than just a narrative.

GitHub profile README

GitHub allows a special repository (username/username) whose README is displayed at the top of your GitHub profile. A striking code image as the hero section of this README makes your profile instantly memorable. Most GitHub profiles are just a list of repositories — a well-designed hero image with a code snippet puts you in the top 5% immediately.

## Hi, I'm Anmol 👋 <p align="center"> <img src="./hero-snippet.png" alt="Code snippet showcase" width="650"> </p> ## What I'm building ...

CV and PDF résumé

This is an underused technique. Export a small (400–500px wide) code snippet in a clean light theme and embed it in your CV's header or skills section as a small visual element. Not every recruiter will appreciate this, but the ones who do — typically technical hiring managers — will remember you specifically because no one else does it.

Keep it subtle: a single small image alongside your contact details, not a full-page technical showcase. The message it sends is "I care about craft", not "I put code screenshots in my resume".

The 3-snippet formula for maximum impact

You don't need dozens of code images. Three well-chosen snippets are enough to establish your technical voice. Pick them to show range across these three dimensions:

  • Problem-solving ability: A snippet that shows a non-obvious solution to a real problem. The kind of thing someone would write if they deeply understood the underlying system — not just copy-pasted from Stack Overflow.
  • Code style and craft: A snippet that demonstrates how you naturally write — readable, well-named, well-commented where appropriate. This is your voice as a developer.
  • Domain knowledge: A snippet specific to your area — whether that's a React hook, a database query, a machine learning data transformation, or a systems-level memory management pattern. This signals depth of expertise in your field.

Export all three with the same theme, the same font, and the same background. Visual consistency makes them read as a deliberate collection rather than three random screenshots.

Technical tips for portfolio code images

  • Use light themes for web portfolios: Most portfolio sites have light backgrounds. Light-themed code images integrate more smoothly. If your portfolio has a dark background, match it with a dark theme.
  • Turn on line numbers: Line numbers signal real-code legitimacy. They make the snippet look like it came from an actual file, not a toy example.
  • Set a meaningful filename: Enable the window titlebar and set a descriptive filename like useIntersectionObserver.ts or parse_csv_stream.py. This tells viewers the context before they read a line.
  • Export at 2x: Portfolios often display images at high DPI on Retina displays. Export your snippets at a larger size and let the browser scale them down — they'll look sharper than native-resolution exports.

💡 What not to do: Avoid showing snippets that are obviously copied from tutorials or that demonstrate only very basic concepts. Recruiters who code can tell the difference between a snippet that shows genuine understanding and one that was assembled from documentation. Pick code you actually wrote for a real purpose.

PNG vs JPEG vs PDF vs HTML — Which Format Should You Use?

By Anmol Dubey 5 min read February 2025

CodeHilite lets you export in five formats: PNG, JPEG, PDF, HTML, and plain text. Most of the time you'll reach for PNG without thinking about it, but knowing when to use the others can save you time and produce better results for specific use cases.

PNG — use this by default

PNG (Portable Network Graphics) is a lossless format, meaning it compresses image data without discarding any of it. What you see in the CodeHilite preview is exactly what you get in the file. No blurring of text edges, no colour shifts, no compression artefacts. PNG also supports transparent backgrounds, which is critical if you want to place your snippet on top of a custom background in Figma, Canva, or Photoshop.

Use PNG for: social media posts (Twitter, Instagram, LinkedIn), website and blog embeds, GitHub README files, Notion pages, presentations (PowerPoint/Keynote will preserve the quality), and any context where you want the image to look exactly right every time.

File size: PNG files from CodeHilite are typically 50–300KB depending on the size of the snippet and the complexity of the background. This is well within the file size limits of every major social platform.

⚠️ One important caveat: Twitter internally converts most uploaded images to JPEG/WebP regardless of what you upload, which can introduce minor compression artefacts. This is Twitter's behaviour, not CodeHilite's — exporting as PNG still gives you the best possible starting quality, which survives Twitter's compression better than a JPEG would.

JPEG — only when file size is a hard constraint

JPEG is a lossy format: it achieves smaller file sizes by permanently discarding some image data during compression. For photographs, this compression is nearly invisible to the human eye. For code screenshots, which contain sharp text edges and solid colour blocks, JPEG compression creates visible blurring and colour banding (called "JPEG artefacts") that make text harder to read.

When JPEG might make sense: If you're embedding a code image inside a very large PDF document where total file size matters (e.g. a 100-page technical report), using JPEG for the embedded images can reduce the overall PDF size significantly. You lose some sharpness, but at small sizes (under 400px wide) it may be acceptable.

When to absolutely avoid JPEG: Social media posts, any context where the image will be zoomed, dark-background code images (compression artefacts show most visibly on dark backgrounds), and any image with text smaller than 14px.

PDF — for documents and print

The PDF export renders your code snippet as a vector-backed PDF page, which means it scales to any print size without quality loss. If your snippet is destined for an academic paper, a technical report, a printed tutorial handout, or a client proposal, PDF export is the right choice.

PDF export also preserves your code's visual appearance exactly as it appears in the preview — themes, fonts, padding, and all — and renders crisply on laser printers and in high-DPI PDF viewers like Adobe Reader and Preview on macOS.

LinkedIn carousel posts: LinkedIn supports native PDF uploads as "document posts", which display as swipeable card carousels. Create a multi-slide visual presentation with CodeHilite exports — each slide a different snippet — and upload as a PDF. This format consistently gets higher engagement than standard image posts on LinkedIn because it encourages interaction (swiping through slides).

HTML — for web embedding

The HTML export produces a self-contained HTML file that includes the syntax-highlighted code using highlight.js. The code in the HTML output is selectable and copyable text — unlike a PNG, where the code is "baked in" as pixels. This matters in two specific scenarios:

  • Embedding live, interactive code blocks on your website: If you have a personal blog or portfolio site and want to embed a code block that visitors can select and copy, the HTML export gives you a ready-made component that matches your CodeHilite aesthetic.
  • Accessibility: Screen readers can read the actual code text from an HTML block; they can't extract meaning from a PNG image. If accessibility matters for your audience, prefer HTML over PNG.

The HTML export does not include any JavaScript dependencies — all styling is inline CSS — so it can be dropped into any webpage without worrying about conflicts with your existing libraries.

Plain text — for clean code sharing

Plain text export strips all formatting and gives you just the raw code content. Use this when you want to paste the code into a forum, a Slack message, a GitHub comment, or anywhere that already applies its own syntax highlighting. It's also useful for quickly copying a cleaned-up version of code that you've formatted in the editor.

CodeHilite vs Carbon vs Ray.so — Honest Comparison

By Anmol Dubey 9 min read March 2025

There are several tools in the "code to image" space. Carbon has been around the longest and is probably the most well-known. Ray.so from Raycast has excellent default aesthetics. Codeimg.io and Showcode.app round out the main alternatives. If you're deciding which tool to use — or whether to switch to CodeHilite — here's a detailed, honest comparison.

Carbon (carbon.now.sh)

Carbon was one of the first code-screenshot tools and popularised the concept. It has a large user base and the "Carbon" aesthetic — dark background, macOS titlebar, signature gradient backgrounds — became so widespread that many developers associate it with the entire category.

What Carbon does well: It's fast, reliable, and widely understood. Its URL-sharing feature is excellent — you can share a Carbon snippet URL and anyone can open and edit it. The interface is minimal and opinionated, which means less decision paralysis if you just want to quickly generate a screenshot.

Where Carbon falls short: Carbon only exports PNG (via image/png from the DOM). There's no PDF, no HTML export, no plain text. There's no line highlighting feature. The font selection is limited. The tool hasn't seen significant updates in years, and some features (like certain theme combinations) produce inconsistent results. Most importantly, Carbon requires you to be online — there's no offline fallback.

Ray.so

Ray.so was built by Raycast and reflects that team's taste for polish and minimalism. The default output looks genuinely beautiful out of the box — arguably the most aesthetically refined of all the tools by default. If you paste code into Ray.so without changing anything, the result looks great.

What Ray.so does well: Stunning default aesthetics. A curated set of gradient backgrounds. Simple, uncluttered interface. The output quality is excellent and the defaults work well for most use cases without customisation.

Where Ray.so falls short: The customisation options are intentionally limited — this is a deliberate design choice, but it means you can't deviate much from their aesthetic. There's no line highlighting, no file upload, no multiple export formats, and no snippet saving. It's designed for quick, beautiful output, not for a power user who needs control over every detail.

CodeHilite — detailed feature breakdown

CodeHilite was built to address the gaps in both Carbon and Ray.so: more export formats, more customisation options, file upload support, line highlighting, and snippet saving — all in a single tool that runs entirely in your browser with no account required.

Export formats

This is where CodeHilite's advantage is most concrete. Carbon exports PNG only. Ray.so exports PNG only. CodeHilite exports PNG, JPEG, PDF, HTML, and plain text. If you've ever needed to embed a code snippet in a Word document, a PDF report, or a webpage with selectable text, you'll understand immediately why having multiple export formats matters.

Line highlighting

CodeHilite allows you to highlight specific lines with a background tint — an indispensable feature for tutorial creators and educators. When you're explaining code in a blog post or video, being able to visually call out "this is the line that matters" is enormously useful. Neither Carbon nor Ray.so offer this. Codeimg.io has a basic version but CodeHilite's implementation is cleaner.

File upload

Drag and drop a source file (e.g. server.ts, styles.css, requirements.txt) onto the CodeHilite editor and it auto-detects the language and populates the editor. This is faster than copy-pasting from your IDE, especially for files you're sharing directly from your project. No other major code-screenshot tool offers this.

Snippet saving

Within a session, CodeHilite lets you save multiple snippets and switch between them. This is useful when you're creating a series of screenshots for a tutorial — you can set up all your snippets, then export them one by one with consistent settings rather than pasting and clearing repeatedly.

Privacy

All three tools process code in the browser and don't transmit it to a server. However, Carbon and Ray.so both connect to external services for URL sharing and analytics. CodeHilite is entirely self-contained — no analytics, no tracking, no URL sharing service. Your code never leaves your browser session, period. This matters for developers working with proprietary codebases or under NDA.

How to Use Code Images in Tutorials and Online Courses

By Anmol Dubey 8 min read March 2025

Why code images beat IDE screenshots for teaching

The instinct when creating a tutorial is to screenshot your IDE — it's fast and it shows exactly what you're looking at. But IDE screenshots come with significant downsides that compound as your content scales up:

  • Visual noise: IDE screenshots include file trees, toolbars, status bars, tab rows, search panels, and notification popups. Every one of these elements competes with the actual code for the viewer's attention.
  • Platform and version inconsistency: Your VS Code theme is different from your students' VS Code themes. Your file tree structure exposes details you may not want to share. Your extensions affect how the IDE looks. Screenshots taken six months apart with an updated IDE look different from earlier ones.
  • Low contrast in video: IDE screenshots at standard resolution compress badly in YouTube's encoding. Text can become blurry or difficult to read, especially for viewers on mobile. A code image with a larger font size and a clean, high-contrast theme remains readable even at 480p.
  • No annotation control: IDE screenshots make it hard to draw attention to a specific part of the code without adding arrows or boxes in a separate editing step. CodeHilite's line highlighting feature does this natively.

Code images solve all of these problems. They're visually focused, consistent, high-contrast, and give you precise control over what the viewer sees.

How to use line highlighting effectively

Line highlighting — where specific lines get a background tint — is one of CodeHilite's most powerful features for educational content, and one of the most underused. Done well, it transforms a passive code image into an active learning tool.

The principle is simple: highlight the lines you're talking about in the current step of the explanation. If you're walking through an algorithm step by step, create a separate code image for each step with different lines highlighted. The rest of the code is visible (showing context) but the highlighted lines draw the eye immediately.

Practically, this means creating a series of code images from the same snippet — one with lines 1–3 highlighted (for step 1), another with lines 5–8 highlighted (for step 2), and so on. This is much more work than a single screenshot but dramatically increases comprehension for learners.

💡 Progressive disclosure technique: Start with the full snippet, then show it again with specific lines highlighted. This dual-image approach ("here's the whole thing, now let's zoom in on this part") mirrors how good textbooks teach — overview first, then detail. The first image gives learners a mental model of the whole; the highlighted image focuses their attention without losing the context.

A practical content creation workflow

Ad hoc screenshot generation gets messy as your content library grows. After creating dozens of tutorials, you end up with a folder of inconsistently named PNG files that are hard to manage. Here's a workflow that scales:

  1. Prepare your code first. Write and test the code in your IDE. Only screenshot it when it's final — re-exporting because you fixed a typo wastes time.
  2. Set your standard settings profile. Before exporting, confirm your theme, font size, card width, and background match your course's established style. CodeHilite remembers your last settings within a session.
  3. Use descriptive filenames when saving. Enable the window titlebar in CodeHilite and set a filename that describes the snippet (step3-render-function.tsx). The exported PNG filename will match, making your assets folder organised.
  4. Export all variants at once. If you need highlighted versions, export the base image first, then create highlighted versions with the same settings. Group related files with a prefix: step3-base.png, step3-highlighted-lines-8-12.png.
  5. Store assets by lesson, not by type. Keep all images for Lesson 4 in a lesson-04/ folder alongside the video and script, not in a global images/ folder. This makes it much easier to find and update specific assets later.

Tips specifically for YouTube and video courses

Code images in video need to handle compression differently from static images. YouTube encodes at variable bitrate, and dark regions with text can suffer from blockiness (macroblocking) at lower quality settings. A few techniques help:

  • Upload in 1080p minimum. At 1080p, YouTube's encoder allocates more bitrate to preserve text sharpness. 720p can work, but 1080p is the safe baseline for code content.
  • Use themes with moderate contrast. Extremely dark backgrounds with very bright text (high contrast ratio) actually compresses worse in some codecs than a moderately dark background with slightly less bright text. Atom One Dark and Nord compress better than pure-black themes.
  • Spend enough time on each slide. Viewers need time to scan code. A good rule of thumb: leave the code image on screen for at least 1 second per line shown. A 20-line snippet should be visible for at least 20 seconds before you move on.
  • Add code images to your video thumbnail. YouTube thumbnails with code snippets visible (partially) get higher click-through rates from developer audiences because they signal technical content immediately. Show 3–5 lines of recognisable code alongside your title text.

Tips specifically for written tutorials and blog posts

In written tutorials, code images should be used selectively. Long-form technical articles that use only code images (with no copyable code blocks) frustrate readers who want to copy and paste to follow along. The best practice is a hybrid approach:

  • Use code images at the beginning of each section as a visual introduction to what you're about to explain.
  • Follow each code image with a copyable code block (a standard HTML <pre><code> block) for readers who want to use the code directly.
  • Reserve code images for key architectural decisions and the "final" version of code; use standard code blocks for intermediate steps.

This approach gives you the visual quality of code images for the parts that matter most, while preserving the practical utility of copyable text for readers following along.

Building a consistent visual brand

If you create content regularly — whether on YouTube, a blog, Twitter, or a course platform — visual consistency across your code images is one of the cheapest ways to build a recognisable brand. When viewers see a code screenshot with your consistent aesthetic, they should be able to identify it as yours before they see your name.

Pick a combination and commit: one dark theme, one light theme (for blog posts), one font, one standard padding level, one background style. Document these settings somewhere and apply them every time. The combination you choose doesn't matter much — what matters is that you're consistent.
